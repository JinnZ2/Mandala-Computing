## 1. Inconsistencies

### 1.1 Two unrelated glyph sets both claim to represent the 8 octahedral states

- **Canonical set** (`⊕⊖⊗⊘⊙⊚⊛⊜`, i.e. U+2295–U+229C): defined in `octahedral_arithmetic.py:30`,
  `mandala_computer.py:38` (`_STATE_GLYPHS`), `quantum_mandala.py:29` (`_STATE_GLYPHS`), and the
  canonical schema file `glyphs/mandala.json` / `atlas/glyphs/mandala.json`
  (`"octahedral_state_glyphs"`). This is also the set shown in `ONBOARDING.md`'s state table
  (lines 42–52) and in `README.md`'s worked example (`'glyph_pair': ('⊘⊖', '⊚⊖')`, line 23).
- **Unrelated set** (`◈◉◊○●◐◑◒`): defined independently in `mandala_computing_explorer.py:21`
  as `GLYPHS = ['◈', '◉', '◊', '○', '●', '◐', '◑', '◒']`, used to render the *same* 8 octahedral
  cell states in that file's mandala visualization panel.
- **Fix:** import the canonical glyphs instead of redefining them:
  ```python
  # mandala_computing_explorer.py, replace line 21
  from octahedral_arithmetic import GLYPHS
  ```
  (or read `glyphs/mandala.json` directly, the way `quantum_mandala.py` does at lines 31–38).

### 1.2 Module count and test count are inconsistent across the doc set

`CLAUDE.md` (lines 467, 722) states the test suite has **336 tests** across a much larger module
set — and this is the *current, correct* number (verified this session: `python tests/test_core.py`
→ "336 passed, 0 failed"). But the following files all still say **"45 tests"** / **"10 modules"**:

| File | Line | Claim |
|---|---|---|
| `README.md` | 30, 211 | "10 Python modules, 45 tests" |
| `README.md` | 169 | `# Run tests (45 tests)` |
| `Integration.md` | 15 | "45 tests passing, factors N=221" |
| `Mandala-octahedral.md` | 22 | "45 tests verify the software stack" |
| `P=np-hypothesis.md` | 111 | `# Full test suite (45 tests)` |

- **Fix:** these five files were clearly written at an earlier snapshot of the repo (9-10 modules,
  45 tests) and never updated as the module count grew toward the current ~26 root-level `.py`
  files / 336 tests. Either update the numbers to match `CLAUDE.md`, or better: have exactly one
  source of truth (`CLAUDE.md`) and have the others say "see CLAUDE.md for current module/test
  counts" instead of hardcoding a number that will drift again.

### 1.3 `ONBOARDING.md`'s module table is a stale subset presented as the full picture

`ONBOARDING.md:190` states **"Six Python modules organized into three dependency layers"** and
lists only `octahedral_arithmetic.py`, `glyph_convert.py`, `mandala_simulator.py`,
`mandala_computer.py`, `holographic_mandala.py`, `quantum_mandala.py`. `CLAUDE.md`'s own
dependency-layers table (lines 72–85) lists 10 layers and roughly 18 core modules, not 6. The
heading reads as an exhaustive inventory rather than "the modules needed for this onboarding
path," which is presumably the intent.

- **Fix:** reword the heading to `## Modules on This Learning Path` (or similar) so it's explicit
  that it's a curated subset, not the full module list — CLAUDE.md already owns the full list.

## 2. Markdown Information Gaps

### 2.1 `consciousness_metrics.py` is referenced as a real, working file — it isn't (see 1.5)

Both `Mandala_integration.md:324` and `Quantum_integration.md:318` present a fully-formed
`compute_phi()` function as "Built into simulator." A reader (human or AI) has no way to tell,
from either file alone, that this is speculative/illustrative pseudocode rather than a pointer to
real source. **Likely intent:** the author was sketching what a future consciousness-metrics
module *would* look like, using the same "here's the code" style used elsewhere in these docs for
things that *are* implemented — the speculative framing just didn't make it into the text the way
it did in `README.md`'s "What This Does NOT Do" section.

### 2.2 Cross-repo file references are inconsistently attributed

`P=np-hypothesis.md:129` and `:141` correctly tag their non-local references: *"Source:
`Mandala/theoretical_validator.py` **in the GeoBin repo**."* and *"From
`experiments/factorization_landscape.py` **in GeoBin**:"*. This is the right pattern. But
`CLAUDE.md:316`'s reference to *"the generic `scale_invariance_breakdown.py` epistemology
pattern (CC0)"* never says where that generic (non-`mandala_`-prefixed) file actually lives —
it isn't in this repo (confirmed: no file named `scale_invariance_breakdown.py` exists here, only
`mandala_scale_invariance_breakdown.py`). **Likely intent:** same CC0 "shared pattern, per-repo
instantiation" convention as `claim_schema.py`, but the source-of-truth location was left
unstated.

- **Suggested fix:** adopt the `P=np-hypothesis.md` convention repo-wide — every reference to a
  file that isn't in this repo gets an explicit `(external: RepoName)` tag.

### 2.3 `Checklist.md` and `Integration.md` give contradictory completion percentages for what reads like the same integration effort

`Checklist.md:370` states *"Overall integration: ~60% complete"* with *"To reach 90%: Need to
implement the missing adapter classes"* (i.e. 90% is a **future** target). `Integration.md:333`
states the *same* integration is already at **90%** with the row explicitly labeled *"✅ Mission
Accomplished."* Neither file is dated, and neither references the other's percentage, so a reader
can't tell whether `Integration.md` is a later, updated snapshot of `Checklist.md`'s roadmap, or
whether they're describing different scopes. **Likely intent:** `Integration.md` probably *is* the
"after" state that `Checklist.md`'s roadmap produced — but nothing says so explicitly.

- **Suggested fix:** add a one-line note at the top of `Integration.md`: *"Supersedes the
  60%→90% roadmap in Checklist.md — this document reflects the completed state."*

### 2.4 No `CHANGELOG.md` despite an explicit versioning scheme in module docstrings

Module docstrings use `v1.0`/`v2.0` (e.g. `mandala_computer.py`'s "MANDALA COMPUTING SIMULATOR
v2.0", `quantum_mandala.py`'s "v2.0: JSON layer wiring, QAOA, sensor telemetry, introspection",
`osl.py`'s "Octahedral Symbolic Language v1.0"), but there is no `CHANGELOG.md` anywhere explaining
what changed between v1.0 and v2.0, or listing what "v3.0" of `.fieldlink.json` changed relative
to earlier versions. A new reader has no way to know what "v2.0" means without diffing git history
themselves.

- **Suggested fix:** a lightweight `CHANGELOG.md` at root, even just one bullet list per module
  version bump, would resolve this and also help the "Discoverability" goals in Section 6.

### 2.5 `README.md`'s "Quick Start" never mentions `experiments/`

The Quick Start section (`README.md:155-174`) lists demo commands for every root engine module
but never mentions `experiments/constant_swapping_simulator.py`, `geometric_computation_selector.py`,
or `mandala_computing_explorer.py` — all three of which are runnable, Jupyter-oriented
playgrounds documented in `experiments/README.md`. A newcomer reading only the top-level README
would not discover these exist.

- **Suggested fix:** add one line: *"See `experiments/README.md` for interactive Jupyter
  playgrounds (Ising explorer, solver selector, constant-swap experiment search)."*

### 2.6 `Checklist.md` and `Integration.md` both embed large blocks of speculative Python inside markdown with no "this is pseudocode" marker

Files like `Checklist.md:132-232` and `Questions.md` (nearly its entire body) present multi-class
Python snippets in fenced code blocks with no explicit "illustrative / not shipped" label the way
`Integration.md`'s "conceptual" tag does elsewhere. Because these render exactly like real,
copy-pasteable source (and *do* use real import-style syntax), a reader skimming rather than
reading closely could reasonably try to `pip install` or import from them and be confused when
nothing exists. **Likely intent:** these are design sketches supporting the prose, not shipped
code — same root cause as 2.1 and 1.4.

- **Suggested fix:** a repo-wide convention — e.g., prefix speculative code fences with an
  HTML comment `<!-- SKETCH: not implemented -->` or a leading `# SKETCH — not shipped in this repo`
  comment inside the fence — would let both humans and crawlers/LLMs distinguish "this is what
  exists" from "this is what we're imagining" at a glance.

### 2.7 No top-level "Getting Started for Contributors" doc

Every doc in the repo is either user-facing (`README.md`), architecture-facing (`CLAUDE.md`), or
theory/vision-facing (the remaining 15 `.md` files). None of them cover contributor mechanics:
how to run just one module's tests, coding-style expectations beyond the table in `CLAUDE.md`,
or how new "schema module" files (`claim_schema.py`-style) are expected to register themselves
(e.g. should a new one be added to `mandala.claims`/`CLAIM_TABLE.json`?). For a project explicitly
inviting contributions (`README.md:205`: *"Contributions welcome"*), there's no `CONTRIBUTING.md`.

## 3. Code Audit

### 3.1 New engine methods added this session have no test coverage

`tests/test_core.py` imports and exercises 20 modules (verified via its own `import` lines) but
never imports `geometric_computation_selector.py`, `mandala_computing_explorer.py`,
`experiments/constant_swapping_simulator.py`, `curiosity_engine.py`, `claim_schema.py`,
`mandala_computing_module.py`, or `mandala_scale_invariance_breakdown.py`. More specifically, the
three new `QuantumMandalaComputer` methods added this session —
`transverse_field_ising_anneal()`, `trotter_suzuki_qmc()`, and the module-level
`export_ising_qiskit()` in `quantum_mandala.py` — were only verified with ad hoc manual scripts
during development, not added to `tests/test_core.py`. Since this file *is* covered by the test
suite for its other methods, these three are a coverage gap in an otherwise well-tested module.

- **Suggested fix:** add to `tests/test_core.py` (pattern-matching the existing quantum tests
  around line 232):
  ```python
  def test_transverse_field_ising_anneal_lowers_energy():
      qc = QuantumMandalaComputer(golden_depth=1, sacred_geometry=8)
      J = np.zeros((3, 3)); h = np.array([1.0, -1.0, 0.5])
      result = qc.transverse_field_ising_anneal(J, h, steps=30)
      assert result["final_energy"] <= result["history"]["energy"][0]

  def test_trotter_suzuki_qmc_returns_valid_spin_config():
      qc = QuantumMandalaComputer(golden_depth=1, sacred_geometry=8)
      J = np.zeros((2, 2)); h = np.array([1.0, -1.0])
      result = qc.trotter_suzuki_qmc(J, h, trotter_steps=5, sweeps=50)
      assert set(result["spin_config"]) <= {-1, 1}
  ```

### 3.2 Bare `except Exception` blocks swallow errors with no diagnostic trace

| File | Line | Context |
|---|---|---|
| `octahedral_resilience.py` | 138 | Heartbeat check failure → marks node `FAILED` |
| `geis.py` | 206 | Token round-trip validation → returns `False` |
| `mandala_runtime.py` | 201 | Stream → Basin projection → silently skipped |
| `mandala_runtime.py` | 1552 | Claim verification fallback → returns `1.0` |

These are all *intentional* per the repo's own documented "breathing degrades, never fails"
philosophy (`CLAUDE.md`'s key-invariants), so none of these are bugs to fix by removing the
`except`. But none of them log *what* went wrong, which makes silent degradation
indistinguishable from silent bugs during debugging.

- **Suggested fix** (minimal, behavior-preserving):
  ```python
  except Exception as e:
      logging.debug("stream %r failed to project to basin: %s", stream, e)
      continue
  ```

### 3.3 `BenchmarkCollector` in `geometric_computation_selector.py` does a full file read+write on every single benchmark call

`geometric_computation_selector.py:227-248`: `add_benchmark()` calls `self.save()` on every
invocation, and `save()` (line 244) re-serializes the *entire* `METHODS` registry's accumulated
benchmark history to `benchmark_db.json` on every call, unconditionally. For interactive/notebook
use this is harmless, but if `run_benchmark()` is ever called in a tight loop (e.g. from
`ParameterSweep`-style code, or a future automated tuning script), this becomes O(n) file I/O per
sample. It's also not safe for concurrent use — two processes writing `benchmark_db.json`
simultaneously can corrupt each other's write (no file locking, no atomic rename).

- **Suggested fix:** batch saves (e.g. save every N calls, or on an explicit `.flush()`/context-manager
  exit), and use the standard atomic-write pattern (write to a temp file, then `os.replace()`) if
  concurrent use becomes a real scenario.

### 3.4 `mandala_computing_module.py`'s hardcoded repo stats are already stale

`mandala_computing_module.py:40-42` hardcodes `REPO_COMMITS = 107`, `TEST_COUNT = 336`,
`MODULE_COUNT = 18`. As of this review, `git rev-list --count HEAD` returns **113** — six commits
past what the file claims, accumulated in the time since those constants were last updated earlier
in this same session. This is the same class of problem as Section 1.2, but self-inflicted by
design: any hardcoded repo-statistics constant with no update mechanism will always be stale by
the time it's read.

- **Suggested fix:** either compute `REPO_COMMITS` at import time (`subprocess.run(["git",
  "rev-list", "--count", "HEAD"], ...)`, with a static fallback if not in a git checkout), or
  relabel the constant/docstring to make clear it's a snapshot as of a specific date rather than a
  live count.

### 3.5 No CI/CD configured at all

There is no `.github/` directory (confirmed: `ls .github` → no such file or directory), so the
336-test suite never runs automatically on push or PR. Nothing currently prevents a change from
silently breaking `tests/test_core.py` before it's merged.

- **Suggested fix:** a minimal `.github/workflows/tests.yml`:
  ```yaml
  name: tests
  on: [push, pull_request]
  jobs:
    test:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/checkout@v4
        - uses: actions/setup-python@v5
          with: { python-version: "3.11" }
        - run: pip install numpy scipy
        - run: python tests/test_core.py
  ```

### 3.6 Test suite is pytest-compatible in structure but not run with pytest

`tests/test_core.py`'s 336 `test_*()` functions all use bare `assert` statements with no
`unittest.TestCase` classes and no custom fixtures — this is exactly the shape pytest expects for
auto-discovery. But the file is invoked as `python tests/test_core.py`, driven by a hand-rolled
runner at the bottom of the file (lines ~3860-3883: iterate `globals()` for `test_` prefixes,
print PASS/FAIL, `sys.exit(1)` on failure). This means none of pytest's tooling — `-k` filters,
`--tb=short`, parallel execution via `pytest-xdist`, coverage via `pytest-cov`, IDE test-runner
integration — is available, despite the tests already being written in the compatible style.

- **Suggested fix:** this is a very low-cost win — `pip install pytest && pytest tests/test_core.py`
  should work today with no code changes (the custom `if __name__ == "__main__":` runner block at
  the bottom is simply ignored by pytest's collection). Worth verifying and then documenting
  `pytest tests/` as the recommended invocation, alongside (not necessarily instead of) the
  existing `python tests/test_core.py`.

### 3.7 Unpinned dependencies

`requirements.txt` and `examples/requirements.txt` both list bare `numpy` / `scipy` with no
version constraints. Reasonable for a research repo prioritizing simplicity, but a future
major-version change in either library (e.g. a NumPy 2.x deprecation) could silently break the
suite with no signal until someone happens to run it on a fresh environment.

- **Suggested fix:** at minimum pin a tested-floor version, e.g. `numpy>=1.24`, `scipy>=1.10`,
  without over-constraining the ceiling.

### 3.8 `examples/requirements.txt` is a verbatim duplicate of the root `requirements.txt`

Both files contain exactly `numpy` and `scipy`, nothing else. If a new example ever needs an
additional dependency, there are now two files to remember to update, and they can silently drift
out of sync.

- **Suggested fix:** delete `examples/requirements.txt` and have any docs that reference it point
  to the root file instead, unless examples are expected to need dependencies the core engine
  doesn't (in which case, keep it but have it read `-r ../requirements.txt` plus example-only
  extras).

### 3.9 `SeedSplitter`'s security-critical randomness is correctly separated from simulation randomness — worth calling out as a positive, with one documentation gap

`octahedral_resilience.py`'s `SeedSplitter.split()` (lines 275-297) correctly uses
`secrets.token_bytes()` (cryptographically secure) for Shamir secret-sharing polynomial
coefficients, while `random.random()` is used only for non-security simulated behavior (heartbeat
failure injection, simulated verification success rates at lines 98, 102, 343, 621). This is the
*right* separation and is worth preserving carefully in any future edits to this file — a
regression here (e.g. someone "simplifying" by switching `secrets.token_bytes` to `random` for
consistency with the rest of the file) would silently break the security guarantee. The one gap:
the modulus `P = (1 << 128) + 51` is commented as *"Next prime after 2^128"* (line 271) with no
citation or in-repo verification; this review independently verified it via Miller-Rabin (confirmed
prime), but a one-line `assert`-backed test would make that guarantee self-documenting rather than
resting on an unverified comment.

## 4. Organizational Structure Suggestions

1. **Move theory/vision docs into `docs/`.** 15 of the 17 root `.md` files
   (`Math.md`, `P=np-hypothesis.md`, `Quantum_integration.md`, `Hardware.md`,
   `Physical-computer.md`, `Bridge-substrate.md`, `Consumer-hardware.md`,
   `Mandala-octahedral.md`, `Mandala_integration.md`, `Checklist.md`, `Questions.md`,
   `Integration.md`, `Notes.md`, `ONBOARDING.md`, `tests/firmware_specification.md`) are
   theory/vision/status documents, not user-facing entry points. Keeping only `README.md`,
   `CLAUDE.md`, `LICENSE`, and `PROJECTS.md` at root would make the top-level listing scannable,
   and moving the rest to `docs/` is also a prerequisite for cheaply enabling GitHub Pages later
   (see Section 6). This is a pure file-move (`git mv *.md docs/`, keeping the four exceptions),
   with a follow-up pass to fix relative links.

2. **Split `mandala_runtime.py` (2639 lines) into a package.** This is already flagged as a known
   risk in `CLAUDE.md`'s own risk-assessment table (#6, "Large module sizes... Acceptable for
   research code; would need splitting before production use"). `CLAUDE.md`'s own module
   documentation already itemizes the natural seams: `Substrate`/`StreamCapability` (stream
   protocol), `Basin`/`Manifest`/`UnifiedGeometry` (data model), the three
   `*IntersectionRule` classes (domain logic), `DynamicsProjector` and its subclasses (LID
   bridge), and `MandalaRuntime` itself (orchestrator). Turning this into
   `mandala_runtime/{substrate,basin,intersection_rules,projectors,runtime}.py` with a
   `mandala_runtime/__init__.py` re-exporting the current public names would be a
   backward-compatible split (`from mandala_runtime import MandalaRuntime` keeps working) that
   makes a solo maintainer's "where do I add X" question answerable from the directory listing
   alone.

3. **Split `octahedral_resilience.py` (1551 lines) the same way**, along its own already-documented
   class list (`HeartbeatMonitor`, `OctahedralCluster`, `SeedSplitter`/`SeedDispersal`,
   `CircuitBreaker`, `AuditTrail`, `FencingManager`, `ShareMerkleTree`) — same rationale as #2, and
   the same low-risk backward-compatible package conversion.

4. **Split `tests/test_core.py` (3883 lines, 336 tests) into one file per source module.** Given the
   file is already pytest-shaped (Section 3.6), splitting into `tests/test_mandala_computer.py`,
   `tests/test_quantum_mandala.py`, `tests/test_octahedral_arithmetic.py`, etc. — mirroring the
   module list — would let a contributor run `pytest tests/test_quantum_mandala.py` in isolation
   instead of always executing all 336 tests, and would make "which module's tests failed" visible
   from the file list in CI output rather than only from scrolling PASS/FAIL output.

5. **Adopt a minimal `pyproject.toml`.** There is currently no package metadata at all — the repo
   only works if `import mandala_computer` is run with the repo root on `sys.path` (which is why
   `examples/*.py` all manually do `sys.path.insert(0, ...)`, per the pattern noted in this
   session's own work on `experiments/constant_swapping_simulator.py`). A minimal
   `pyproject.toml`:
   ```toml
   [project]
   name = "mandala-computing"
   version = "2.0.0"
   dependencies = ["numpy>=1.24", "scipy>=1.10"]

   [tool.setuptools]
   py-modules = ["mandala_computer", "quantum_mandala", "holographic_mandala", "..."]
   ```
   would let `pip install -e .` work from any directory, removing the need for manual `sys.path`
   surgery in every subdirectory script, and is also a discoverability prerequisite (PyPI listing,
   `pip install mandala-computing`).

6. **Group the CC0 "schema"/self-description modules into their own subpackage.**
   `claim_schema.py`, `mandala_computing_module.py`, `mandala_scale_invariance_breakdown.py`, and
   `curiosity_engine.py` form a coherent family — all CC0-licensed, all "describe this repo to an
   AI reader" utilities, none of them part of the computational engine. They currently sit
   undifferentiated at root next to `mandala_computer.py`/`quantum_mandala.py`. A subpackage like
   `epistemics/` (or `schema/`) would let a newcomer immediately distinguish "the thing that
   computes" from "the thing that describes the repo to an AI."

7. **Add a sync-check for the duplicated glyph JSON files.** `glyphs/mandala.json` and
   `atlas/glyphs/mandala.json` are byte-identical duplicates, kept as two separate files only
   because `glyphs/mandala.json` is an external contract path consumed by BioGrid2.0 (confirmed

   def test_glyph_json_files_stay_in_sync():

   breaking the external contract.

8. **Give `experiments/` a matching `examples/`-style discoverability entry.** `CLAUDE.md`'s

   `examples/*.py` file, but `experiments/*.py` and the two root-level playgrounds

   Since `experiments/README.md` already documents what each one does and how it's wired, the fix
   is just a one-line pointer from `ONBOARDING.md`'s "What's Next" section (already suggested in

## 5. Limitations Mitigation Checklist

Assessed against the actual implementation in `claim_validator.py`, `claim_schema.py`, and
related modules — not against the docs' aspirational framing.

### 5.1 Symbolic–Subsymbolic Gap — **Partially addressed**

`claim_validator.py:225-239` (`extract_relations()`) does regex-based subject/predicate/object
extraction (`POSITIVE_PRED`/`NEGATIVE_PRED` at lines 197-206) — a very light form of logical-form
extraction, but not formal predicate logic, and not connected to any symbolic solver (no SAT/SMT/
theorem-prover integration exists anywhere in the codebase). Separately,
`geometric_state_algebra.py` implements real symbolic machinery (O_h group, Cayley graph, group
ring algebra) — but that's for the *computation* side of the repo (solving problems), not for
claim grounding/verification; the two are not connected.

- **Recommendation:** the repo already has a working SAT solver
  (`MandalaComputer.encode_sat()`/`simulated_annealing()`). A natural, low-effort next step:
  encode `extract_relations()`'s subject/object pairs as a small constraint-satisfaction instance
  (each contradiction = an unsatisfiable clause) and route it through the *existing* SAT engine
  instead of the current O(pairs²) contradiction scan in `check_consistency()`
  (`claim_validator.py:242-268`) — genuinely dogfooding the repo's own computation engine on its
  own epistemology module.

### 5.2 Grounding Problem — **Partially addressed**

- Units/dimensions: **missing.** No `check_units`, `dimensional_analysis`, or SI-unit validation
  exists anywhere in the codebase (verified by repo-wide grep).
- Lower-layer constraints enforced: **addressed.** `claim_validator.py:416-419` implements a real
  tier-floor: *"if t1_score > 0.6: weighted_score = max(weighted_score, t1_score)"* — a physics/
  thermodynamics-tier violation genuinely floors the overall concern score regardless of how well
  a claim scores on Systems/Empirical tiers, matching the documented "higher tiers cannot override
  lower tier violations" design.
- Meta-grounding flag for revolutionary claims: **partially addressed.** No dedicated flag exists,
  but `claim_validator.py:413-414`'s `multi_boost = 0.1 * max(0, n_flagged - 2)` functions
  similarly — a claim tripping concern across many tiers simultaneously (the signature of an
  extraordinary/revolutionary claim) gets an explicit score boost.
- **Recommendation:** add a `check_units(text)` pass using a small SI-unit token regex (kg, J, m/s,
  W, mol, K, ...) that flags claims combining units in physically nonsensical ways (e.g. adding a
  mass to an energy) as a 6th validation dimension alongside the existing five.

### 5.3 Semantic Ambiguity — **Partially addressed**

- Vague terms quantified: **addressed.** `VAGUE_QUANTIFIERS` (line 121) and `TEMPORAL_VAGUE`
  (line 134) explicitly enumerate and count vague terms ("many", "most", "significant",
  "always", "inevitably", ...).
- Scope explicit (temporal/spatial/ontological): **addressed for structured claims only.**
  `claim_schema.py`'s `Claim` dataclass has explicit `bounds`/`cond` fields (e.g.
  `MANDALA_CLAIMS[0]`'s `bounds=["octahedral_lattice", "8_states", "0-7"]`), but raw
  natural-language text passed directly to `validate_claim()` gets no scope extraction — only
  vague-word counting.
- Reference class specified: **missing.** No mechanism anywhere detects or requires a stated
  comparison baseline ("compared to X", "relative to Y").
- **Recommendation:** add a `REFERENCE_CLASS` regex (`compared to|relative to|versus|vs\.|baseline
  of`) as a new falsifiability sub-signal, and for prose claims, offer a lightweight scope-extraction
  helper that mirrors `claim_schema.py`'s `bounds`/`cond` fields rather than only being available
  to pre-structured `Claim` objects.

### 5.4 Falsifiability Paradox — **Best-covered item in the checklist; mostly addressed**

- Escape-hatch detector: **addressed.** `UNFALSIFIABLE_FRAMING` (`claim_validator.py:145-150`)
  explicitly catches "essentially", "in principle", "by definition", "self-evidently",
  "axiomatically" — this is a real, working escape-hatch detector, not just a description of one.
- Falsifiable/unfalsifiable classifier: **addressed.** `analyze_falsifiability()` (lines 163-190)
  returns an explicit `FALSIFIABLE` / `PARTIALLY FALSIFIABLE` / `LOW FALSIFIABILITY` classification
  with a numeric score.
- Refutation-observation set enumerable: **addressed for structured claims, missing for prose.**
  `claim_schema.py`'s `Claim.fail` field is exactly this — an explicit, enumerated list of
  falsifying conditions per claim (e.g. `MANDALA_CLAIMS[1]`'s `fail=["T=0_frozen",
  "disconnected_graph"]`). But `validate_claim()` on raw prose only *scores* falsifiability; it
  doesn't generate a candidate refutation set the way the structured `Claim` format does.
- **Recommendation:** the one real gap is generalizing `Claim.fail`-style enumeration to
  arbitrary prose — e.g. after `extract_relations()`, auto-generate a candidate falsifier per
  extracted relation ("would be refuted by observing {object} NOT {inverse-of-direction}
  {subject}").

### 5.5 Formal Verification vs. Complexity — **Partially addressed**

- Formal proof scoped: **missing.** `Math.md`'s "theorems" are hand-written prose proofs; there is
  no Z3/Lean/Coq integration or any machine-checked proof anywhere in the repo.
- Background knowledge accessible: **addressed.** `CLAIM_TABLE.json` + `mandala.claims` /
  `mandala.claims.bin` (per `claim_schema.py`'s four-layer format) is exactly a compact, versioned,
  accessible background-knowledge store — confirmed consistent (12 claims in `mandala.claims`
  match the 12 `Claim` entries in `claim_schema.py`'s `MANDALA_CLAIMS`).
- Probabilistic fallback with confidence: **addressed.** `validate_claim()`'s `overall_concern`
  score with `LOW`/`MODERATE`/`HIGH`/`VERY HIGH CONCERN` bands (lines 423-430) is precisely a
  confidence-scored fallback in place of formal proof.
- **Recommendation:** since formal proof is explicitly out of scope for a "concern score" system,
  the main improvement available is documenting *why* — a short section in `claim_validator.py`'s
  module docstring explicitly stating "this is a heuristic concern score, not a formal proof
  system; treat `overall_concern` as a triage signal, not a verdict" would make the scoping
  decision legible to a caller who might otherwise over-trust the number.
